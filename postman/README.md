🚀 성능 테스트 결과
📌 개요
대용량 동시 예약 요청 처리를 위한 기존 낙관적 락 방식과 Redis 기반 분산 락 + 큐 아키텍처의 성능 비교 테스트
🧪 테스트 환경

총 요청 수: 50,000건
요청 방식: HTTP POST
테스트 도구: Postman Collection Runner
엔드포인트: /room/advancedBulkTest
테스트 파라미터:

방 개수: 10개
배치 처리: 활성화



📊 성능 비교
처리 방식총 소요 시간TPS요청당 처리 시간특징낙관적 락~58분 40초~144,167ms대량 요청 시 심각한 병목 현상Redis + 큐14.3초3,4980.29ms대규모 트래픽에도 안정적 처리
📈 성능 개선 효과

처리 속도: 245배 향상
처리량(TPS): 250배 증가
확장성: 동시 요청 증가에도 선형적 성능 유지

🏗️ 아키텍처 비교
기존: 낙관적 락 방식
클라이언트 → API → DB (충돌 시 재시도)

❌ 인기 자원에 대한 높은 경합
❌ 재시도 오버헤드 기하급수적 증가
❌ 데이터베이스 병목 현상

개선: Redis 기반 큐 아키텍처
클라이언트 → API → Redis 큐 → 병렬 워커 → DB

✅ 분산 락으로 레이스 컨디션 방지
✅ 큐 기반 처리로 워커 수 조절 가능
✅ 배치 처리로 DB 작업 최소화
✅ 비동기 처리로 응답 속도 개선

✅ 결론
Redis 기반 큐 구조와 병렬 처리 도입으로 대규모 동시 요청 처리 능력이 획기적으로 개선되었으며, 실제 운영 환경에서도 안정적인 서비스 제공이 가능한 수준으로 성능이 향상됨.
